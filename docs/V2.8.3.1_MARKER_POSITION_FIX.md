## 问题描述

用户报告图表显示异常：**所有K线图形都挤在左侧，买卖点标记都在右侧，位置完全对不上**。

### 问题截图分析

观察用户提供的 `macd_chart.png`:
- **左侧**: K线、成交量、MACD 等所有数据挤在一起
- **右侧**: 7个买入标记（红色▲）和7个卖出标记（绿色▼）分散排列
- **问题**: 标记点与实际交易发生的K线位置完全不对应

### 示例

```
图表布局（错误）:
|K线K线K线K线K线...                                    ▲▼▲▼▲▼▲|
|左侧挤满了数据                              右侧是买卖点标记|
|                                                            |

应该是（正确）:
|K线..▲..K线K线..▼..K线..▲..K线K线..▼..K线..▲..K线K线..▼..|
|买卖点标记应该出现在对应的K线位置                         |
```

## 根本原因分析

### Backtrader 的 x 轴坐标系统

Backtrader 的 plot 功能使用**数值索引**作为 x 轴坐标：
- 第1根K线: x = 0
- 第2根K线: x = 1
- 第3根K线: x = 2
- ...
- 第N根K线: x = N-1

这是一个**连续的整数序列**，与实际日期无关。

### 我们的错误实现

在 V2.8.3 中，我们使用 `datetime` 对象作为 scatter 的 x 坐标：

```python
# 错误的实现
buy_dates = []  # [datetime(2024,3,13), datetime(2024,4,24), ...]
buy_prices = []  # [1740.84, 1697.83, ...]

price_ax.scatter(
    buy_dates,      # ❌ datetime 对象
    buy_prices,
    marker='^',
    ...
)
```

**问题**:
- Matplotlib 将 `datetime` 对象自动转换为浮点数（距离某个基准日期的天数）
- 这些浮点数（如 738000+）远大于 Backtrader 的索引范围（0-241）
- 导致标记点全部绘制在图表的**最右侧**（超出数据范围）

### 坐标系不匹配示意

```
Backtrader x 轴: [0, 1, 2, 3, ..., 240, 241]
                 └─ K线位置

datetime 转换:   [738950, 738978, 739002, ...]
                 └─ 远超出K线范围，全部在右侧
```

---

## 解决方案

### 核心思路

**构建日期到索引的映射表**，将买卖点的 `datetime` 转换为 Backtrader 的数值索引。

### 实现步骤

#### 1. 构建日期到索引映射

```python
date_to_index = {}
data_len = len(data)

# 从后往前遍历（因为 Backtrader 使用负索引访问历史数据）
for i in range(data_len):
    try:
        # 使用负索引访问: -1 是最后一根K线，-2 是倒数第二根...
        date_num = data.datetime[-i-1]
        date_obj = bt.num2date(date_num)
        date_key = date_obj.date()
        
        # 存储正向索引（从0开始）
        date_to_index[date_key] = data_len - i - 1
    except:
        continue
```

**映射表示例**:
```python
{
    date(2024, 1, 2):  0,
    date(2024, 1, 3):  1,
    date(2024, 1, 4):  2,
    ...
    date(2024, 3, 13): 46,   # 第一笔买入
    date(2024, 3, 21): 52,   # 第一笔卖出
    ...
    date(2024, 12, 26): 241  # 最后一根K线
}
```

#### 2. 转换买卖点日期为索引

```python
buy_indices = []
buy_prices = []

for order in strat._orders:
    if order.status == order.Completed and order.isbuy():
        exec_date = bt.num2date(order.executed.dt).date()  # datetime.date
        price = order.executed.price
        
        # 关键：使用映射表查找索引
        if exec_date in date_to_index:
            idx = date_to_index[exec_date]
            buy_indices.append(idx)
            buy_prices.append(price)
```

**转换示例**:
```python
# 输入（订单日期）
exec_date = date(2024, 3, 13)

# 查找映射
idx = date_to_index[date(2024, 3, 13)]  # = 46

# 输出（数值索引）
buy_indices = [46, 74, 121, ...]  # ✅ 与 Backtrader x 轴对齐
```

#### 3. 使用索引绘制标记

```python
# 正确的实现
price_ax.scatter(
    buy_indices,    # ✅ 数值索引 [46, 74, 121, ...]
    buy_prices,     # [1740.84, 1697.83, ...]
    marker='^',
    color='red',
    s=200,
    ...
)
```

## 技术细节

### Backtrader 数据访问模式

```python
# Backtrader 使用负索引访问历史数据
data.close[0]    # 当前K线的收盘价
data.close[-1]   # 上一根K线的收盘价
data.close[-2]   # 倒数第二根K线的收盘价

# 对于已完成的回测，所有数据都可以通过负索引访问
data.datetime[-i-1]  # 倒数第 i+1 根K线的时间戳
```

### 日期转换流程

```
1. Backtrader 内部时间戳 (float)
   ↓ bt.num2date()
2. Python datetime 对象
   ↓ .date()
3. Python date 对象 (yyyy-mm-dd)
   ↓ 映射表查找
4. 数值索引 (int: 0, 1, 2, ...)
   ↓ scatter()
5. 绘制在正确位置
```

### 为什么需要负索引遍历

Backtrader 的数据存储是**动态的**，在策略运行时逐步累积：
- 策略第1天: data 只有 1 个值
- 策略第2天: data 有 2 个值
- ...
- 策略结束: data 有 N 个值

使用**负索引 `-i-1`** 可以从最新数据向前遍历，确保访问所有历史数据。

### 边界条件处理

```python
try:
    date_num = data.datetime[-i-1]
    # ...
except:
    continue  # 跳过无效数据点
```

某些情况下可能访问超出范围的数据（如数据缺失、停牌日），使用 try-except 确保程序不会崩溃。

## 常见问题

### Q1: 为什么之前使用 datetime 可以显示，只是位置不对？

**A**: Matplotlib 的 scatter 方法非常灵活，可以接受多种类型的 x 坐标：
- 整数/浮点数: 直接使用
- datetime 对象: 自动转换为浮点数（matplotlib date number）

问题是 Backtrader plot 使用的是**整数索引系统**，而 datetime 转换后是**大浮点数**，两者不在同一坐标系。

### Q2: 如果交易日期不在数据中怎么办？

**A**: 使用 `if exec_date in date_to_index` 检查，只添加存在于映射表中的日期。如果交易日期缺失（如停牌日），该标记会被跳过。

### Q3: 能否直接修改 Backtrader 的坐标系？

**A**: 不推荐。Backtrader 的整数索引系统是内部设计，强制改为日期坐标会破坏：
- 技术指标的绘制
- 子图的对齐
- 缩放和交互功能

更好的方法是**适配 Backtrader 的坐标系**。

### Q4: 如何验证标记位置是否正确？

**A**: 
1. 查看控制台输出的交易日志，记录交易日期
2. 在图表中查找对应日期的K线
3. 检查该位置是否有买卖点标记
4. 标记的价格应该与交易执行价格一致

示例验证：
```
交易日志: 2024-03-13, BUY EXECUTED, Price: 1740.84
图表查看: 3月13日的K线上方有红色▲，价格约为1740
结论: ✅ 位置正确
```

## 性能影响

### 时间复杂度

- 构建映射表: O(N)，N = K线数量（通常 < 500）
- 转换买卖点: O(M)，M = 交易次数（通常 < 50）
- **总计**: O(N + M)，可忽略不计

### 内存使用

- 映射表: ~8KB (假设 250 个交易日 × 32 bytes)
- 索引列表: ~1KB (假设 50 笔交易 × 16 bytes)
- **总计**: < 10KB，完全可接受
